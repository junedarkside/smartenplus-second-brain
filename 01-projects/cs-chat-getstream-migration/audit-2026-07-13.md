# CS Chat → GetStream Migration — Audit Report

> **STATUS:** AUDIT-COMPLETE · **VERDICT: CONDITIONAL** · 2026-07-13 (session #242)
>
> 6-agent Workflow fan-out (329k tokens, 470s): 3 code-ref verifiers (BE/FE/AD) + GetStream tier/cost web + PDPA/legal web + adversarial synthesizer. Read-only. The corrections below were applied to [[implementation-plan-2026-07-13]] · [[cs-chat-getstream-hybrid-2026-07-13]] · README · [[getstream-migration-debate-2026-07-13]] in the same session.

## Summary

**CONDITIONAL — not PASS, not FAIL.** The ADR's two load-bearing claims survive adversarial scrutiny: GetStream **Build tier is genuinely free** ($0, 1,000 MAU, 100 concurrent, no credit card, indefinite retention — all confirmed on getstream.io/chat/pricing), and **PDPA is a soft-block resolvable before Phase 1** (Singapore region selectable on Build at provisioning, published DPA, `deleteUsers`/`exportUser` erasure APIs exist). Backend architecture is sound; the schema assumption (Conversation/Message lack `transport`/`stream_id`/`custom_data`) is **confirmed correct**; migration 0015 is genuinely additive; sender-integrity enum matches exactly.

**However**, the plan carries a cluster of high-severity inaccuracies that must close before any code ships: (1) the **FeatureFlag cutover mechanism is impossible as specified** (model has no `value` field), (2) a silent **system-message status-transition break** in admin-dashboard, (3) the **"no staff inbox today" value-prop is factually false** (MUI inbox exists at `pages/cs/index.js`), (4) **pricing-table structural errors** (Build is hard-capped, not paid-overage; Start = $499/mo not $399), and a **Python 3.9 ↔ SDK requires ≥3.10 conflict** that adds undocumented effort.

## Blockers (must-close before Phase 1)

### B1 — FeatureFlag cutover impossible as specified · HIGH
`FeatureFlag` model (`cs/models.py:128-140`) has only `key/enabled/updated_by/updated_at` — **no `value` field**. Plan Phase 5 Task 2 `FeatureFlag(key='cs_chat_transport', value='stream'|'legacy')` cannot work. `FeatureFlagView` (`views.py:725`) only accepts `PATCH {enabled: bool}`. Also: **`cs_chat` FeatureFlag does not exist in backend** — it is FE-only, so the rollback "kill switch" doesn't exist as described.
**Resolution:** (a) migration adding `FeatureFlag.value` CharField (nullable, coexist with `enabled`), or (b) dedicated `CsTransportConfig` model, or (c) settings/env `NEXT_PUBLIC_CHAT_TRANSPORT` (already referenced in Phase 2/5). Pick one before Phase 5 design freezes.

### B2 — System-message status-transition realtime path lost in admin-dashboard · HIGH
`useStaffInboxRealtime` is mounted **globally in `pages/dashboard/SideList.js:116`** (NOT `pages/cs/index.js`) and is load-bearing for: closed-conv auto-reopen mirroring, **system-message status transitions (reopen/close/ended) that have no other realtime path**, and OS Web Push triggers. Stream's `notification.message_new` fires only for customer messages, NOT Django-originated system status rows. Treating this hook as a simple source swap **silently breaks status-transition propagation**.
**Resolution:** Phase 3 Task 4 must either (a) keep a Supabase/Django subscription for SYSTEM messages alongside Stream, or (b) re-route status transitions through a Stream system channel. Add a dedicated risk row.

### B3 — "No staff inbox today" value-prop is false · MEDIUM
ADR:57 / debate:77-78 / plan:490 assert the migration "ships staff inbox for free" because none exists. A **fully-functional MUI inbox EXISTS at `pages/cs/index.js` (185 lines)**: master-detail layout, status filters, search, unread badges, push opt-in, realtime unread updates, mark-read-on-focus, conversation merge, OTA deep-links. This was a verdict-changer driving STAY→MIGRATE reversal and is partially illusory.
**Resolution:** Reframe Phase 3 goal as "swap realtime source under existing inbox + add Stream presence/typing", not "build staff inbox". Re-evaluate cost/effort justification.

### B4 — Pricing-table structural errors invalidate cost framing · MEDIUM
Build and Maker are **not two parallel free tiers** — Build is the free dev tier (hard-capped "Limits Apply"); Maker is a separate eligibility-gated free ACCOUNT type ($100/mo credit, <5 team + <$10k/mo). SmartEnPlus uses Build; Maker is a red herring. The **overage rates $0.09/user + $0.99/concurrent belong to Start PAID tier, NOT Build** (Build is hard-capped — exceeding rate-limits/errors, no paid overage). **Start = $499/mo ($5,988/yr list), not $399** — the debate's ~$4.8k/yr understates the real cost curve ($0 → step to $499/mo, no smooth overage).
**Resolution:** Collapse Build/Maker rows; reframe Build as hard service-degradation (not predictable paid overage); re-anchor OPEX to $499/mo Start list.

## Corrections applied this session

Applied to the plan/ADR/README/debate in the same session (see those docs for in-place edits):

**Stale/inaccurate line refs**
- `SupabaseTokenView`: `990-1025` → **`1334-1406`** (plan Phase 1 Task 3, ADR Critical files, README Critical files)
- README helpers `cs/views.py:1-50` → **`:62 (_resolve_guest_token), :426 (_resolve_message_sender)`**

**Backend design corrections**
- `os.environ.get('STREAM_API_KEY')` → **`config('STREAM_API_KEY', default='')`** (python-decouple, match `settings.py:595-598`)
- Python "3.11" → **3.9** (`python:3.9-alpine`); **add Phase-0 task: verify `getstream` installs on 3.9, else bump Python (undocumented effort)**
- `getstream==<5.x>` → **`getstream==3.5.*`** (PyPI 3.5.0, `requires_python >=3.10,<4.0` — no 5.x line)
- `mint_stream_jwt(user_id, scope, conv_id, ttl)`: `mint_supabase_jwt` takes a **claims dict** (TTL 900s), not positional args; use **getstream SDK `client.create_token(user_id, expiration)`** directly; note 24h TTL ≠ supabase 15min
- `ConversationListSerializer/DetailSerializer` → **`ConversationSerializer` (`serializers.py:10`) + `ConversationInboxSerializer` (`:17`)**; add `stream_id` to `MessageSerializer.Meta.fields` (`:84`)
- `cs_throttles.cs_otp` → **`CsPollThrottle` (`cs/throttles.py:8`)** to mirror `SupabaseTokenView` throttle (`views.py:1337`)
- `sync_chat_messages` reads **FROM Supabase→Django** (gap-fill); `insert_cs_message` writes Django→Supabase on send — plan conflated the two directions

**Frontend design corrections**
- ChatWidget branch: current is **2-way `USE_REALTIME` boolean** (`ChatWidget.js:175-176`), not 3-way transport-typed. A `stream` transport requires adding `transport` to conversation API response + `SET_CONV` action + deriving flags from `state.transport` (does not exist today)
- Text-send path: plan only describes image divergence — add task to **replace the `else if (USE_REALTIME)` Supabase-direct-insert branch (`ChatPanel.js:68-81`)** with a Stream `sendMessage` call; AD side remove `supabase.from('cs_messages').insert` staff-send branch (`ConversationDetail.js:134-148`)
- `useGetStreamChat` need not mirror `useChatPolling`'s `cursor` param — Stream SDK owns cursor/watch state internally

**Admin-dashboard corrections**
- AD CS module is **MUI sx-prop styled, NOT Tailwind** (zero `className=` in `components/cs/`+`pages/cs/`). `StreamConversationDetail.js` must reimplement bubble in MUI, not reuse FE's Tailwind bubble. Drop AD portion of "Tailwind override" risk; add real risk: stream-chat-react CSS-in-JS vs MUI CssBaseline/theme
- Staff send today is **bimodal**: realtime mode = direct Supabase insert bypassing Django (`ConversationDetail.js:110-153`); only polling mode uses RTK mutation. Stream replacement must remove the insert + ensure webhook archives so polling fallback/history stay consistent
- `useStaffInboxRealtime` mount = `SideList.js:116` (global), NOT `pages/cs/index.js` — preserve on swap
- AD deliberately uses **isolated per-hook Supabase clients** (not shared singleton) to prevent token clobbering; verify Stream `StreamChat.getInstance()` singleton doesn't race staff token refresh between the two concurrent hooks

**Pain-point freshness (ADR §Context)**
- #1 reconnect/gap-fill: **shipped** (`88e845de`, `useChatRealtime.js:144-174`) — "flaky" unverified statically
- #2 token-refresh silent fail: refresh-path **fixed** (`useChatRealtime.js:61-69` → `onConversationClosed`); initial-mint-path **still silent** (`:77-80`); real BE failure point is `upsert_cs_conversation` 502, not JWT mint

- #6 history display broken: **FE fixed** (`useChatRealtime.js:91-115` paginates Django); **AD works** (`ConversationDetail.js:44-47` + `useStaffChatRealtime.js:91-114`) — narrow or drop this pain

**Vendor/PDPA corrections**
