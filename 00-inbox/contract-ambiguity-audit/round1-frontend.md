## What Works
- The category matrix in `serviceCategoryHelper.js:17-99` plus `CATEGORY_CAPABILITIES` in `components/contracts/constants.js:91-102` is the only place render-gating logic is centralized. Day-trip vs transport is mutually exclusive on the customer side, with no overlap.
- `MeetingPointCard` correctly handles the `meeting_point_type` null case (MeetingPointCard.js:8 — `if (!meetingPointType) return null`).
- `ActivitySearch` debounces search text (300ms) and skips initial mount to avoid clearing URL state (ActivitySearch.js:35-38).
- Admin help text for `primaryLocation` (line 182-186) explicitly tells staff it is **not** the search index — better than expected for a system with this much ambiguity.
- `dayTripsApi.getActivityLocations` cache is 1 hour, `getStationSuggestions` is 15 min — appropriate for low-volatility location data.

## Open Questions
- **Q1.** Does the backend `info_fields` schema *ever* return `field_type` in camelCase? Passengers.js:259-275 has a switch with no `default: trip[someField] = ''` — the entire field is silently dropped. If backend ever ships `pickupPoint` (camelCase) the form stops working without an error.
- **Q2.** Is `service_areas` actually the search facet, or is it a denormalised side-effect of the `/locations/` endpoint reading from a different table? Admin help text claims it is the index; without backend audit, the indirection is unverified.
- **Q3.** Are there 1+ customer journeys that expose both `MeetingPointCard` (read-only) and a free-text pickup input on the same booking? Current reading: no — `MeetingPointCard` is on tour detail pages, free-text pickup is on transport checkout. But the contract model allows both fields to be populated.
- **Q4.** `BookingDetail.js.backup` (legacy file) at `components/bookings/BookingDetail.js.backup:346` still references the same `info_fields.pickuppoint` shape — is the `.backup` file in active use or dead? If active, the casing shim exists in 5 places not 4.
- **Q5.** The admin `ContractFormFields.js:300-305` passes `data={formik.values.dndCharacterData?.place}` to `TimelineDisplay` (admin) — but the customer `BookingDetail/ServiceDetail.js:182-186` passes `data={timeline}` where `timeline = contract?.timeline?.timeline_place` (`ServiceDetail.js:29`). Two different property paths to the same shape. Is the admin DnD timeline the *source* for the customer's read-only render, or are they maintained independently?
- **Q6.** `getActivityLocations` accepts a `service_category` filter but the customer never sets it (ActivitySearch.js:40-42 only passes `filters.category`). The endpoint is hard-coded to `DAY_TOUR` as default. So multi-day tours and spa/accommodation activities will never appear in the autocomplete regardless of `service_areas` configuration.
- **Q7.** The prop `serviceCategory={filters.category}` in `FilterDayTripsPage.js:48` and `serviceCategory` passed to `getContracts` (line 47) — if the filter is `MULTI_DAY_TOUR`, the day-trip list query returns it but the autocomplete (line 41) sends `undefined` and falls back to `DAY_TOUR` default.
